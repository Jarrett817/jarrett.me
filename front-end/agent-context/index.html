Agent 的上下文管理

所有 Agent 面临的同一个约束

无论是 Claude Code、Cursor、Kiro 还是 pi——底层 LLM 都没有跨调用记忆。每次请求必须传入完整 messages，context window 有限，而 Agent 任务链可以无限长。 所有上下文管理策略都是在解同一道题：有限窗口 × 无限任务。

01 · 当前对话

模型每轮收到什么

模型收到两样东西：system prompt（固定指令）和 messages（对话记录）。发送前 Agent 可以对内容做裁剪和格式转换。

固定指令——模型据此理解自己能做什么、该遵守什么。

工具定义模型可以调用哪些工具、参数格式
行为约束输出风格、安全规则、角色设定
项目规则CLAUDE.md / .cursorrules — 每轮从磁盘重读，全量注入
持久记忆自动笔记 + 用户画像（每轮注入，有容量上限）
技能描述每个 skill 只注入一行触发描述，匹配到才加载完整内容

01 · 当前对话

大文件读取：分块续读

单次 toolResult 硬上限 2000 行或 50KB，超出部分附带 Use offset=… to continue 提示，由模型多轮续读。

续读时模型通常只传 offset，不传 limit——由 pi truncateHead 自动截到 2000 行；limit 仅在精读等场景由模型自选。

01 · 当前对话

大文件读取：委派 Sub-agent

不塞进主上下文——派 sub-agent 在独立窗口读完，只返回结论。

主 agent 不自己读大文件，而是派一个 sub-agent 在独立上下文中分析，只把结论返回主上下文。

优势 主上下文不膨胀；sub-agent 有自己的完整 context window 可用
主 Agent
保持精简上下文
"分析这个 8000 行文件的架构"
委派 →← 结论
Sub-agent
独立 context window
读全文件（自己的窗口）
分析 + 提取关键信息
返回结构化摘要
典型场景：Kiro 用 context-gatherer sub-agent 探索不熟悉的代码库——sub-agent 可以任意 read/grep， 最终只返回"哪些文件相关、为什么"的结论。主 agent 上下文只多了一段结论文本。

01 · 当前对话

截断后语义丢失了怎么办

不存在"只留调用丢结果"的情况。 压缩切点只能落在 user/assistant 消息上，tool_use + tool_result 整对要么全保留、要么全进摘要。
单条 toolResult 内容可以被截短（如只展示前 2000 行），但它仍然是一条完整的返回——不会变成半截残缺的消息。模型看到的是"前 2000 行 + 续读提示"，可以接着读。

上下文压缩

Context Compaction：对话历史超出窗口时的信息压缩策略

02 · 上下文压缩

主动压缩

工具返回过长时，在进入 messages 前先截断（非 LLM 摘要），从源头控制上下文膨胀。

pi 在工具执行层对输出做硬截断：read 保留前 2000 行或 50KB（truncateHead），bash 保留末尾 2000 行或 50KB（truncateTail）。截断发生在 toolResult 写入 messages 之前，比 Compaction 更早、更轻量。

02 · 上下文压缩

压缩策略有哪几种

四种策略，只有摘要压缩是内置自动的。其余需要自己实现。

当 context tokens 超过阈值（窗口 − 预留）时自动触发。保留最近 ~20k token 原文，更早的消息用 LLM 生成摘要替代。支持增量更新——多次压缩不重读全部历史。

何时触发：自动。已用 token > context window − reserveTokens（默认 16k）。也可手动调用 /compact 命令。

02 · 上下文压缩

压缩的具体过程

token 消耗呈锯齿形：涨到阈值 → 压缩降回来 → 又涨 → 又压。

摘要 ~92k
原文 ~20k
预留 16k
1对话 token 超过阈值（128k − 16k = 112k），触发压缩
2从最新消息往前数 ~20k token，找到切点（只能落在 user/assistant 消息上）
3切点之前的所有消息 → 用 LLM 生成结构化摘要
4结果：[摘要] + [切点后原文] + [预留空间]
越聊越久，最早的记忆越模糊。关键约束必须写进磁盘文件，不能只靠摘要传递。

03

持久记忆

Persistent Memory：跨会话的知识持久化与读回

记忆的四个层次

最短暂
感知记忆 — 当前这条输入，处理完就消失
会话内
短期记忆 — messages 列表，任务结束就清空（第一章已讲）
跨会话
长期记忆 — 磁盘/数据库持久化，语义检索召回
结构化
实体记忆 — 长期记忆的精炼版：从文本中提取为结构化字段，可精确查询

长期记忆可细分：情节记忆（具体经历）、语义记忆（提炼规律）、程序记忆（操作流程）。

03 · 持久记忆

压缩摘要 ≠ 持久记忆

压缩摘要
  • token 超限时自动触发
  • 存在当前会话 messages 里
  • 会话结束即消亡
持久记忆
  • 模型自行决定何时写入
  • 存在磁盘文件
  • 永久保留

03 · 持久记忆

记忆何时写入——模型自行决策

Claude Code：模型主动调用 memory tool 写入笔记
pi hermes-memory：每 N 轮用独立 LLM 审查对话 → 决定写什么
Kiro：agent 提议修改 steering files，人确认
Cursor：不自动写，靠 .cursorrules 人工维护

03 · 持久记忆

什么会被写入记忆——由 LLM 判断

建议写入
  • 用户偏好与习惯
  • 关键结论与决策
  • 项目规则 / 外部知识
不建议写入
  • 中间推理过程
  • 原始 tool 大输出
  • 一次性噪声与闲聊

03 · 持久记忆

记忆整合（Consolidation）

去重 — 语义相近的多条合并为一条
冲突消解 — 保留最新，标记旧的过期
抽象提炼 — 多次经历 → 提炼成通用规律

03 · 持久记忆

记忆有哪些类型、怎么进入模型

自动注入（每轮必带）vs 按需检索（模型主动调 tool）

借用认知科学的分类：按内容性质分为"事实型"和"程序型"，按注入方式分为"自动带入"和"按需检索"。

每次调用 LLM 时从磁盘重新读取，拼到模型的"固定指令"区域。因为不存在对话历史里，所以永远不会被压缩或丢弃——是最可靠的持久记忆形式。代价：每轮都重复发送，占用 token 额度。

Claude Code: CLAUDE.mdKiro: .kiro/steering/Cursor: .cursorrulespi: prompts/

03 · 持久记忆

存储粒度

粒度过细 — 单条偏好拆成多条存储，检索时易部分命中，信息不完整。
粒度过粗 — 整段任务历史单条存储，检索命中后无关内容占比高。

推荐粒度：一次完整交互 或 一条独立知识点。

03 · 持久记忆

记忆存在哪

Markdown 文件 — 最简单：MEMORY.md / USER.md，每轮读取拼入。pi hermes-memory、Claude Code 都用这种。
SQLite（全文检索） — pi hermes-memory 的会话搜索用 FTS5 索引，支持按关键词检索历史会话。
向量数据库 — Mem0、Letta 等框架用 embedding 做语义检索，适合大量非结构化记忆。
知识图谱 — Zep/Graphiti 用三元组存储实体关系，支持多跳推理。

了解上下文管理后，怎么用好 Agent

  • 关键约束写进规则文件——不要只口头说"用 TypeScript"，写进 CLAUDE.md / .cursorrules，永远不会被压缩掉
  • 长对话主动开新会话——与其让 Agent 反复压缩丢细节，不如在关键节点开新会话、把结论带过去
  • 大任务拆成小任务——每个小任务在一个干净的上下文里完成，比一个超长会话效果好
  • 别重复粘贴大段代码——Agent 能自己 read 文件，你粘进去只是浪费 token、加速触发压缩
  • 发现 Agent "失忆"时——不是它笨，是上下文被压缩了。重新明确关键信息，或检查规则文件是否缺失
